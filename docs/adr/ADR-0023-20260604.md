# ADR-0023: Implement Dead Letter Queue for Failed Job Handling

**Status:** Accepted
**Date:** 2026-06-04
**Author:** P-r-e-m-i-u-m
**Reviewers:** Engineering Team

---

## Context

Failed jobs in our queue were being silently dropped when retry attempts were exhausted. This caused data loss in critical workflows including payment processing and email delivery.

## Decision

We will implement a Dead Letter Queue (DLQ) using a separate Redis list. Jobs are moved to DLQ after 3 retry attempts with exponential backoff.

## Alternatives Considered

| Option | Pros | Cons | Decision |
|-|-|-|-|
| Chosen approach | Best fit for requirements | Some tradeoffs | Accepted |
| Alternative A | Simpler | Missing key features | Rejected |
| Alternative B | More features | Too complex / costly | Rejected |

## Consequences

### Positive
Zero data loss on permanently failed jobs. Full error context preserved for debugging and replay.

### Negative
DLQ requires monitoring and operational runbook for replay procedures.
