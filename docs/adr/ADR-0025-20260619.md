# ADR-0025: Adopt Cursor-Based Pagination Over Offset Pagination

**Status:** Accepted
**Date:** 2026-06-19
**Author:** P-r-e-m-i-u-m
**Reviewers:** Engineering Team

---

## Context

Our list endpoints were using offset-based pagination which caused significant performance degradation as dataset size grew. At 500k records, page 100 queries took 4.2 seconds due to full table scans.

## Decision

We will implement cursor-based pagination using a composite cursor (id + created_at) encoded as base64. Offset pagination will be deprecated and removed in v2.0.

## Alternatives Considered

| Option | Pros | Cons | Decision |
|-|-|-|-|
| Chosen approach | Best fit for requirements | Some tradeoffs | Accepted |
| Alternative A | Simpler | Missing key features | Rejected |
| Alternative B | More features | Too complex / costly | Rejected |

## Consequences

### Positive
List endpoint p99 latency: constant 12ms regardless of dataset size. Eliminated page drift on high-write collections.

### Negative
Cannot support arbitrary page jumping — linked list navigation only.
