## Summary

This RFC proposes a formal API versioning strategy using URL path versioning with a defined deprecation policy to allow breaking changes without disrupting existing API consumers.

## Motivation

Breaking changes require coordinating all API consumers simultaneously. No formal deprecation process. Enterprise customers require API stability guarantees.

## Detailed Design

URL structure: /v1/users/:id, /v2/users/:id. Deprecation policy: minimum 6 months notice. Deprecated versions return Sunset header. Maximum 2 active major versions at any time.

## Alternatives Considered

Header versioning: cleaner URLs but harder to test in browser. Query param versioning: simple but pollutes query string. No versioning: requires all changes to be additive only.
